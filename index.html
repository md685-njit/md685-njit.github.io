Criterion

An AI-powered coding assignment platform built for CS instructors — automated grading, personalized student feedback, Canvas integration, and a full in-browser code editor.

Open Criterion  →  criteriongrade.com

What makes Criterion different

AI Feedback on Every Submission

Claude AI writes a personalized 300–400 word analysis for each student — not a template. It explains exactly which tests passed, which failed, why, and specific steps to fix the issues. No competing tool does this.

Full In-Browser IDE

Students write, run, and debug code directly in Criterion using the Monaco editor — the same engine powering VS Code. No installation, no environment setup, works on any computer or tablet.

Automated + Human Grading

Test cases run automatically. You review, adjust scores, add inline code comments, and post to Canvas — all from a single three-panel workspace. Rubric grading is built in alongside test scores.

Canvas-Native Integration

Sync your full roster via Canvas. Post grades directly back to Canvas gradebook columns with one click. No CSV exports, no manual entry, no copy-paste.

AI-Generated Test Cases & Rubrics

Describe your assignment and let the AI draft a complete set of test cases and a rubric in seconds. Edit and refine them — AI gives you a structure to react to, saving 30–60 minutes of setup per assignment.

Real-Time Grading Progress

Watch test results stream in live as grading runs — per-student, per-test, in real time via a server-sent events stream. No refresh, no waiting for a batch email.

Quick Start

Platform URL: https://criteriongrade.com

Step 1 — Create Your Account

  1. Go to criteriongrade.com and click Create an account.
  2. Enter your full name and institutional email. Choose a password (8+ characters, at least one letter and one number).
  3. Select Instructor from the role dropdown.
  4. Check your email for the verification link and click it. You'll see a confirmation banner on the login page.
  5. Log in and accept the data usage consent on first login.

Step 2 — Get Your Canvas Access Token

  1. Go to njit.instructure.com and log in.
  2. Click your profile picture in the top-left corner, then go to Account → Settings.
  3. Scroll down to Approved Integrations and click New Access Token.
  4. Give it a name (e.g., Criterion), leave the expiry blank, and click Generate Token.
  5. Copy the token immediately — it is only shown once and cannot be retrieved again.

Step 3 — Connect Canvas in Criterion

  1. In Criterion, look for the Canvas panel in the top-right corner of the instructor dashboard. Click Connect Canvas Account.
  2. A panel slides open. In the domain field, type njit.instructure.com.
  3. Paste the token you copied from Canvas into the text field below the domain.
  4. Click Connect with pasted token. Your Canvas courses appear immediately in the sync panel below.

We are currently using personal Canvas API tokens for testing because we do not yet have university-level credentials to set up the full OAuth integration. Once the university provides those, students will be able to log in directly through Canvas SSO with no extra steps. Until then, the manual token method works identically for all features.

This is 100% safe — connecting your Canvas token gives Criterion read-only access to your course roster and the ability to post grades back to Canvas. No personal information, student data, or existing class grades are modified in any way. The token is encrypted at rest and never exposed outside the platform.

Step 4 — Import a Course

  1. After connecting your token, a list of all your Canvas courses appears on the dashboard. Click Sync Courses from Canvas if it does not load automatically.
  2. You will see all the courses tied to your Canvas account. For testing, pick any one course from the list and check the checkbox next to it.
  3. Click Sync Selected Courses. Criterion imports the course name, roster, sections, and enrollment roles. A live progress log updates as it runs — usually 5–30 seconds.
  4. Once done, the course appears on your dashboard and you are ready to create assignments inside it.

Step 5 — Create a Test Assignment

Criterion supports Python, C, C++, Java, and JavaScript. The example below uses Python — feel free to use whichever language you teach. Just name your starter file with the right extension (e.g., main.c, Main.java, index.js) and Criterion auto-detects the language. You can also leave starter files out entirely if you prefer students to start from scratch.
  1. Click your imported course, then click New Assignment.
  2. On the Basic Info tab, give it a title (e.g., Hello World — Python) and a short description explaining the problem. The description is shown to students inside their workspace every time they open the assignment.
  3. On the Starter Files tab, click New File, name it main.py, and paste a skeleton: 
    def greet(name):
    # TODO: return a greeting string
    pass

print(greet("World"))
  4. On the Test Cases tab, click Generate Test Cases with AI. The AI drafts tests from your description. Review, tweak weights, and save.
  5. Click Save as Draft to save. From the assignment detail page you can also generate a rubric with AI and add a Canvas deep link.

Step 6 — Preview the Student Experience

On the assignment detail page, click View as Student. This opens a fully functional preview workspace — you see exactly what your students see: the description in the sidebar, starter files in the editor, the Run button, and the Submit button. The preview is isolated and does not create real submissions or affect grades.

This is one of the most useful features for assignment design — catch confusing instructions or broken starter files before your students hit them.

Step 7 — Grade a Submission

  1. Have a test student account (or a colleague) click the assignment deep link and submit code.
  2. On the assignment detail page, click Start Grading Session.
  3. Watch real-time progress as each submission is compiled, tested, scored, and receives AI feedback.
  4. Click into a result to open the Grading Workspace — review AI feedback, adjust the score, add inline comments, and click Post to Canvas.
That is the complete loop: Create assignment → student submits → automated grading → AI feedback → instructor reviews → grade posted to Canvas. The rest of this guide covers every feature in depth — use the sidebar to explore.

What makes Criterion different

The biggest gap between Criterion and the other tools is the AI layer. Every submission gets personalized Claude AI feedback — not a generic message, but a 300–400 word analysis of exactly which tests passed, which failed, and concrete steps to fix the failing ones. This feedback is automatically aligned to your rubric when one is active.

The other major difference is the in-browser IDE. Students write, run, and debug code directly in Criterion without installing anything. The Monaco editor (the same engine that powers VS Code) supports syntax highlighting for all five supported languages.

Canvas integration is native rather than bolt-on. Criterion syncs your full roster via OAuth and posts grades directly back to Canvas grade columns — no CSV exports, no manual entry.

Platform Features

Here is a high-level map of what Criterion does. Each feature is covered in depth in its own section — this page is the overview.

Code Execution

Student code is compiled and executed in an isolated sandbox powered by Judge0. Supported languages are Python 3, C (gcc), C++ (g++), Java (OpenJDK), and JavaScript (Node.js). Each execution is stateless, time-limited, and memory-limited. Students can click Run as many times as they want before submitting — runs do not consume attempts.

Test Cases

Criterion supports five types of test cases. You can mix and match them on the same assignment:

  • stdin / stdout — Feed an input, compare the output to an expected string.
  • Unit test — Language-specific test files (pytest, CG_ASSERT, Jest-style, Java).
  • Hidden — Same as stdin/stdout but students only see pass/fail, never the inputs or expected outputs.
  • Script (bash) — A custom shell script that inspects the submission. Exit 0 = pass.
  • Special judge — A Python validator that returns a score fraction (0.0–1.0) for partial credit on problems with multiple valid answers.

AI Feedback

After every submission, Criterion calls the Anthropic Claude API and generates personalized feedback for that specific student. The AI is given the assignment description, your grading instructions (private), the active rubric, the student's source code, and all test results. It produces up to 400 words covering what worked, what failed, and specific next steps. If no API response arrives within 45 seconds, a fallback message is shown and the instructor adds feedback during review.

Rubrics

Rubrics let you grade qualitative dimensions — code quality, design, documentation — alongside automated tests. Each criterion has weighted performance levels with score fractions from 0.0 to 1.0. Criterion AI can draft a complete rubric from your assignment description in one click.

Canvas Integration

Connect Criterion to Canvas via OAuth 2.0. Criterion pulls your course roster, sections, and enrollment roles automatically. When grading is complete, post grades directly to Canvas assignment columns with one click. Canvas tokens are encrypted at rest with Fernet symmetric encryption.

Grading Workflow

Submissions go through a structured review pipeline: Auto-graded → TA reviewed → Instructor confirmed → Posted to Canvas. Every state transition is recorded in the audit log. Instructors can override scores, add inline code comments, and post to Canvas from a single three-panel workspace.

View as Student

Before making an assignment live, click View as Student on the assignment detail page. This opens a fully functional preview workspace — you see exactly what your students will see: the description in the sidebar, starter files pre-loaded in the editor, the Run button, and the Submit button. The preview workspace is isolated — it uses a separate preview copy of the assignment and does not affect real student grades or submissions. Use it to catch broken starter files, confusing instructions, or environment issues before students hit them.

Academic Integrity

Criterion integrates with MOSS (Stanford) for plagiarism detection. Run a check on any assignment and get a ranked list of suspicious pairs with similarity percentages and links to the MOSS side-by-side viewer. Peer review is built in with configurable anonymity modes.

Account Setup

This page walks through registering as an instructor, verifying your email, and completing the first-time consent step.

Registering

  1. Go to criteriongrade.com and click Create an account.
  2. Enter your full name and institutional email address.
  3. Create a password. Requirements: at least 8 characters, must include at least one letter and one number.
  4. Select Instructor from the role dropdown.
  5. Submit the form. Criterion sends a verification email to your address immediately.

Email Verification

Open the verification email and click the link. Your account is now active. If you do not receive the email within a few minutes, check your spam folder. The verification link expires after 24 hours — if it expires, you can request a new one from the login page.

Consent

On your first login you will see the Criterion data usage consent page. This is mandatory — you must accept before accessing the platform. The consent document is versioned; if it is updated in the future, you will be prompted to re-accept on your next login.

If you click Decline, your account is deactivated. Contact your administrator to re-enable it.

Password Reset

If you forget your password, click Forgot password? on the login page. Enter your email address and Criterion sends a time-limited reset link. Click the link, enter a new password, and you are back in.

After 5 consecutive failed login attempts, your account is locked for 30 minutes. This protects against brute-force attacks.

Account Security Tips

  • Use a password that is unique to Criterion — do not reuse passwords from other services.
  • Log out when using a shared or public computer. Sessions expire after 24 hours automatically.
  • Your Canvas API token (if connected) is encrypted at rest — Criterion never stores it in plaintext.

Course Management

Courses in Criterion can be imported from Canvas or created manually. Most instructors will import from Canvas since it automatically pulls in their roster.

Connecting to Canvas

There are two ways to connect, depending on your institution's setup:

Option A — Manual API Token (recommended for testing)

  1. In Canvas, go to Account → Settings, scroll to Approved Integrations, and click New Access Token. Name it anything, set no expiry, and click Generate Token. Copy it — it is only shown once.
  2. In Criterion, click Connect Canvas Account on the instructor dashboard.
  3. In the dialog, paste your token into the manual token field and click Connect.
  4. Your Canvas courses appear immediately.
Manual tokens give identical functionality to OAuth — same roster sync, same grade posting. The only difference is students log into Criterion directly rather than through Canvas SSO.

Option B — OAuth 2.0 (full institutional integration)

  1. Your institution's Canvas admin must register Criterion as a Developer Key in Canvas and provide the client ID and secret to your Criterion administrator.
  2. Once configured, click Connect Canvas Account and you are redirected to your institution's Canvas login.
  3. Authorize Criterion and you are returned with your courses listed.
OAuth setup requires coordination with your institution's IT/Canvas admin team. During the testing phase, use the manual token method.

Importing a Course

After connecting, Criterion shows all courses you have access to in Canvas. Check the courses you want to bring in and click Sync Selected Courses. Criterion then:

  • Imports the course name, course code, and section names.
  • Syncs the full student roster with names and emails.
  • Maps enrollment roles — students, TAs, instructors, observers.
  • Shows a real-time progress log as the sync runs. Large rosters can take 15–30 seconds.
Syncing is non-destructive. If a student dropped the course in Canvas and you re-sync, their Criterion enrollment is updated — no data is deleted.

Sync Status

Each course card on your dashboard shows a sync status badge:

  • Healthy — Synced recently, no errors.
  • Stale — Has not synced in over 6 hours.
  • Failed — Last sync attempt had errors. Check your Canvas token.
  • Syncing — Currently in progress.
  • Paused — You disabled auto-sync for this course.
  • Disconnected — No Canvas token on file.

You can manually re-trigger a sync at any time by clicking the sync button on the course card.

  • Students tab — Shows every enrolled student, their section, and progress metrics: started, submitted, graded, posted.

From the Assignments tab you can create a new assignment or click an existing one to open its detail and settings.

Tab 1 — Basic Information

Title

The student-facing name shown in their dashboard and workspace header. Example: Lab 3: Binary Search Tree.

Description

Markdown-formatted problem statement shown in the student's workspace sidebar. Use this to explain the problem, expected behavior, input/output format, and any constraints. Students see this every time they open the assignment.

Grading Instructions

Private notes that students never see. They are passed to the AI grader and visible to instructors, TAs, and graders. Use this to describe edge cases you care about, partial credit rules, or grading philosophy. Example: "Deduct points if the student uses a global variable instead of passing state through function parameters."

Instruction File

Upload a PDF, Word, or text file (e.g., a detailed spec sheet or assignment handout). It appears as a pinned, read-only link in the student workspace. Students cannot modify or delete it.

Tab 2 — Starter Files

Starter files are skeleton code that pre-populate every student's workspace when they first open the assignment. This is useful for providing function signatures, import statements, or boilerplate they should not have to write from scratch.

  • Upload files using the file picker or drag-and-drop.
  • Or click New File, type a filename (e.g., main.py), and write the content directly in the inline Monaco editor.
  • Preview and edit any file's content before saving.
  • Students can modify starter files freely — they get their own copy.
If you update starter files after students have already opened the assignment, existing student workspaces are not updated. Only new workspace creations get the updated files.

Tab 3 — Test Cases

A quick summary of how scoring works:

score = (sum of weights of passing tests) / (sum of all weights) × points_possible

For example, if you have 10 tests each with weight 1 and a student passes 7, their score is 70 out of 100 (assuming 100 points possible).

Test Cases

Test cases define how student submissions are evaluated. Every test case has a weight that controls its share of the total score. The final score formula is:

score = (sum of weights of passing tests) / (sum of all weights) × points_possible

Fields common to all test case types:

  • Name — Label shown in grading feedback. Make it descriptive: "Edge case: empty list input".
  • Weight — Relative importance. Set harder tests higher. Simple sanity checks might be weight 1, edge cases weight 3.
  • Timeout — Per-test execution timeout in seconds (1–60). Defaults to the assignment-level timeout.
  • Visible to student — If unchecked, the test becomes a hidden test (students see only pass/fail, not details).

Select a test case type to see its specific fields:

stdin / stdout — Standard Input/Output

The most common type. Criterion feeds a standard input to the program and compares its stdout to the expected output.

  • Stdin — Text fed to the program via standard input. Type it directly or upload a file for large inputs.
  • Expected stdout — The exact expected output. Type it or upload a file.
  • Float tolerance — Allowable numeric difference for floating-point comparisons. For example, 0.001 allows answers within ±0.001. Leave blank for exact string matching.
  • Command-line arguments — Optional arguments passed when the program is invoked. Useful for programs that accept argv.
Trailing whitespace and trailing newline differences are normalized automatically before comparison. Minor formatting differences will not cause false failures.

Unit Test — Language-Specific Framework Tests

Write test code using the standard framework for each language. Criterion compiles and runs the test file alongside the student's code.

LanguageFramework / FormatKey requirement
PythonpytestTest functions prefixed with test_
CCG_ASSERT macrosCall cg_run_all_tests() inside main()
C++CG_ASSERT macrosCall cg_run_all_tests() inside main()
JavaCgUnitTest classTest methods prefixed with test inside a CgUnitTest class
JavaScriptJest-styledescribe / it / expect; import student code with require('./main.js')

Test file field — Upload or paste the test code. Criterion injects it into the sandbox alongside the student's files at grading time.

Unit tests give students per-assertion pass/fail detail, helping them pinpoint exactly which function is broken rather than just knowing their output was wrong.

Hidden Test — Secret Evaluation Cases

Hidden tests work identically to stdin/stdout tests, but students never see the input, expected output, or their actual output — only whether the test passed or failed. The test name is also hidden.

Fields are identical to stdin/stdout: Stdin, Expected stdout, Float tolerance, Command-line arguments.

Use hidden tests to check edge cases, boundary conditions, and robustness without telegraphing test data to students. This encourages writing defensive code rather than hard-coding for visible tests.

You can mix visible and hidden tests on the same assignment. A common pattern: 70% of points come from visible tests (basic functionality), 30% from hidden tests (edge cases and stress tests).

Script (Bash) — Custom Shell Judge

Write a bash script that has full access to the student's submission directory. The script can run the program, inspect files, check for specific output patterns, or validate side effects. Exit code 0 means the test passes; any non-zero exit means it fails.

#!/bin/bash
# Example: verify the program creates an output file
# and that it contains at least 10 lines
python3 main.py input.txt output.txt

if [ -f output.txt ] && [ $(wc -l < output.txt) -ge 10 ]; then
  exit 0
else
  exit 1
fi

The script is executed in the same directory as the student's submitted files, so you can reference files by name directly.

Special Judge — Partial-Credit Python Validator

For problems where multiple answers are valid (sorting with ties, approximate numeric answers, graph problems with multiple optimal paths), write a Python 3 checker that awards a score fraction between 0.0 and 1.0.

import sys

expected_file = sys.argv[1]  # path to expected output
actual_file   = sys.argv[2]  # path to student's actual output
input_file    = sys.argv[3]  # path to the test input

with open(actual_file) as f:
    actual = f.read().strip().split('\n')

with open(expected_file) as f:
    expected = f.read().strip().split('\n')

# Count how many lines match
correct = sum(a == e for a, e in zip(actual, expected))
fraction = correct / len(expected) if expected else 1.0

print(fraction)  # Must print a float in [0.0, 1.0] on stdout

Print a single float on the first line of stdout. That float is multiplied by the test case weight to get the test's score contribution.

Rubrics

Rubrics let you score qualitative dimensions of a submission — code style, design choices, documentation, algorithmic efficiency — alongside the automated test results. The rubric editor lives on the assignment detail page, available after you save the assignment.

Creating a Rubric

  1. Open the assignment detail page and scroll to the Rubric section.
  2. Click Add Criterion. Give it a name (e.g., Code Readability) and a description of what it measures.
  3. Set the criterion's weight. Weights are relative — a criterion with weight 2 counts twice as much as one with weight 1. Criterion normalizes them automatically.
  4. Add 3–5 performance levels for this criterion. Each level has:
    • Label — e.g., Excellent, Satisfactory, Needs Work, Not Attempted.
    • Score fraction — A number from 0.0 to 1.0. Excellent might be 1.0, Satisfactory 0.7, Needs Work 0.3, Not Attempted 0.0.
    • Description — What does this level look like? Be specific so graders apply it consistently.
  5. Repeat for each criterion.
  6. Save and mark the rubric as active to apply it to grading.

How the Rubric Score Is Calculated

rubric_score = sum(criterion.weight × chosen_level.fraction) / sum(all weights)
points       = rubric_score × assignment.points_possible

For example: two criteria, weights 1 and 2. Grader selects 1.0 and 0.5 respectively. Rubric score = (1×1.0 + 2×0.5) / 3 = 0.667. On a 100-point assignment that is 66.7 points.

AI-Assisted Rubric Generation

Click Generate Rubric with AI. The AI reads your assignment title, description, and grading instructions, then drafts a complete rubric with criteria, weights, and performance levels. Edit the draft before saving — the AI gives you a structure to react to, not a finished grading policy.

You can also highlight a single criterion and click Suggest levels to have the AI draft performance levels for that one criterion only.

Rubric Versioning

Every time you save a rubric with changes, Criterion creates a new version. You can view previous versions from a dropdown and mark any version as active. Grading sessions record which rubric version was active at the time, so historical results remain accurate even if you later update the rubric.

How Rubric Scoring Appears in the Grading Workspace

When reviewing a submission, the grading workspace shows a rubric panel. For each criterion, the grader selects a performance level from a dropdown and optionally adds a short justification note. The rubric score updates in real time as selections are made. The final score shown to the student reflects the rubric evaluation.

Assignment Settings

The Settings tab in the assignment form controls deadlines, scoring, late policies, Canvas linking, and the execution environment. These settings can be changed after the assignment is live — changes take effect immediately for future submissions.

Dates & Scoring

  • Due Date — Timezone-aware deadline. Students see a live countdown in their workspace header.
  • Points Possible — Default 100. Sets the scale for both test scores and rubric scores.
  • Max Attempts — Leave blank for unlimited. Set a number to cap how many times a student can submit. Students see their remaining attempts in the workspace header.

Test Result Visibility

Controls how much test feedback students see after grading completes:

ModeWhat students see
Full FeedbackPass/fail per test, test name, stdin, expected output, their actual output, errors, and AI feedback
Counts OnlyOnly a summary — e.g., "7 of 10 tests passed" — no per-test details
HiddenTotal score only — no test information at all

Submission Policy

  • Latest submission — Only the most recent submission counts. Earlier attempts are ignored.
  • Best effective score — The highest-scoring attempt (after any late penalty) counts. Students can resubmit to improve their grade. This policy is more forgiving and encourages iterative learning.

Late Policy

PolicyBehavior
Hard CutoffNo submissions accepted after the deadline. The Submit button is disabled.
Grace PeriodAllow submissions for N minutes/hours after the deadline with zero penalty. You set the grace period length. Useful for accommodating minor technical issues at submission time.
Penalty Per DayAllow late submissions indefinitely. Deduct a percentage per 24 hours late. Effective score = raw × (1 − days_late × penalty%). Example: 10% per day, submitted 2 days late, raw score 85 → effective score 85 × 0.8 = 68.
Both the raw score and effective score are always shown to the student so they can see exactly how the penalty was applied.

Canvas Linking

Search for the corresponding Canvas assignment by name and select it. Once linked, Criterion can post grades directly to that Canvas gradebook column. A green Canvas linked badge confirms the connection. You can unlink and relink at any time.

TA Permissions

  • Allow TAs to post grades to Canvas — When enabled, TAs can confirm and post grades themselves.
  • When disabled, TA reviews move to a "TA reviewed" state and require instructor confirmation before posting. This gives you a QC checkpoint on TA grading.

Runtime Environment

Override the default execution environment for specialized assignments:

  • Run timeout — Seconds allowed for the student's manual "Run" button. Does not affect grading.
  • Test timeout — Seconds allowed per test case during grading.
  • Memory limit — Maximum memory (MB) per execution.
  • Compile command — Override the default compiler invocation with a custom bash script. Use this when you need custom compiler flags (e.g., -O2 -Wall -Werror).
  • Run command — Override how the compiled binary is invoked.
  • Dependency manifest — Describe packages or libraries the sandbox should have (e.g., numpy, pandas for Python).
  • Hidden support files — Files injected into the sandbox that students cannot see. Use these for grading harness headers, fixture data files, helper libraries, or any file your tests need but students should not have access to.

Running Grading Sessions

A grading session takes all student submissions for an assignment and runs them through the full evaluation pipeline — compilation, test execution, score calculation, and AI feedback generation. You initiate grading manually; it does not happen automatically on submission.

Starting a Batch Grading Session

  1. Open the assignment detail page.
  2. Click Start Grading Session.
  3. Criterion enqueues all pending submissions as background jobs in the job queue.
  4. A progress bar and counter appear: "14 / 47 completed." This updates in real time via a live server-sent events stream — you do not need to refresh.
  5. As results finish, they appear individually in the grading workspace review queue.
You can leave the page while grading runs. Come back later and all completed results will be waiting in the review queue.

What Happens During Grading

1

Snapshot Frozen

The student's workspace files at submission time are saved as an immutable snapshot. Edits made after submission do not affect the grade.

2

Compilation

For compiled languages (C, C++, Java), the code is compiled first. If compilation fails, all remaining tests are immediately marked as failed and the result moves on. The compilation error is shown in the grading workspace and to the student (if visibility allows).

3

Test Execution

Each test case is executed in an isolated Judge0 sandbox. Results — pass/fail, stdout, stderr, execution time (ms), memory (KB) — are recorded per test. Tests run sequentially.

4

Score Calculation

The weighted sum of passing tests is computed and scaled to the assignment's points possible. The result is rounded to 2 decimal places.

5

AI Feedback Generated

Claude AI generates personalized feedback (up to 400 words) covering what passed, what failed, and specific improvement suggestions. This runs in parallel with step 3 where possible.

6

Ready for Review

The result enters the review queue with status Auto-graded. You or a TA can now inspect, adjust, comment, and post to Canvas.

Regrading

  • Regrade all — Re-runs grading for every student on the assignment. Use this if you updated test cases or fixed an environment issue.
  • Regrade individual — Available on a specific student's result in the grading workspace. Re-runs only their submission.
Regrading overwrites the previous auto-grade score. If an instructor had already manually adjusted a score, re-grading resets it to the new automated result. Manual adjustments made after regrading are preserved.

Submitting Work

When you are ready to submit for grading, click Submit in the workspace toolbar. Here is exactly what happens and what to watch for.

Before You Submit

  • Check that your entrypoint file is set correctly (the file that starts your program).
  • Run your code a few times with different inputs to sanity-check it.
  • Check the attempt counter in the header — if your instructor set a limit, make sure you are not about to use your last attempt on code you are unsure about.

The Submission Process

  1. Click Submit. A confirmation dialog appears showing your current attempt count and how many remain (if limited).
  2. Confirm. Criterion immediately captures a snapshot of all your workspace files at this exact moment. Even if you edit files after submitting, the snapshot is what gets graded.
  3. The grading pipeline starts. You can watch progress in the bottom panel — individual test results appear as they complete.
  4. When all tests are done and AI feedback is generated, the full results appear.

Deadlines & Late Submissions

The workspace header always shows the due date and a countdown. What happens after the deadline depends on your instructor's late policy:

  • Hard cutoff — The Submit button is disabled the moment the deadline passes.
  • Grace period — You can still submit for a short window after the deadline (e.g., 1 hour) with no penalty. The header shows how much grace period time remains.
  • Penalty per day — You can submit any time, but each 24-hour period past the deadline reduces your effective score. Both your raw score and effective score are shown.
Each submission uses one attempt. If you have a limited attempt count, plan carefully. The attempt counter in the header shows exactly how many you have used and how many remain.

After Submitting

Your submission enters the grading pipeline. You will see your results — test outcomes, score, and AI feedback — as soon as grading completes (usually within a minute). Your submission status in My Courses updates from blue (in progress) to amber (submitted / under review).

You can continue editing your workspace while grading runs. Your edits after submission do not affect the submitted snapshot. If you want to improve and resubmit (and you have attempts left), make your changes and hit Submit again.

Grader Calibration

Calibration is a tool for verifying that AI-generated grades and human graders are scoring consistently. Without calibration, AI grades might systematically run high or low compared to your own judgment. Calibration lets you measure and correct that bias.

When to Use It

  • At the start of a semester when you first run an assignment type.
  • After updating a rubric significantly.
  • When a TA joins the team and you want to align their scoring to yours.
  • Any time you suspect grades feel off.

Running a Calibration Session

  1. Open the assignment detail page and click Calibration Workspace.
  2. Create a new calibration session. Criterion automatically selects 5–10 diverse submissions as calibration samples — it tries to pick a spread of scores and code styles.
  3. For each sample, review the student's code and the AI-generated score. Enter the score you would give this submission, without reference to the AI score.
  4. After scoring all samples, the system calculates the bias delta — the average difference between your scores and the AI scores. For example, if your scores averaged 5 points higher, the delta is +5.
  5. Choose whether to apply a correction factor to future grading sessions for this assignment. This shifts all future AI scores by the measured delta.

Understanding the Output

  • Bias delta — The average point difference (your score minus AI score). A positive delta means the AI is grading lower than you would. A negative delta means it is grading higher.
  • Standard deviation — How consistent the discrepancy is. A low standard deviation means the AI is consistently off by roughly the same amount; a high one means it is unpredictable.
  • Correction factor — The multiplier or offset to apply to future AI scores. Applying it does not retroactively change past grades.
Calibration history is preserved. You can compare correction factors across sessions to see whether the AI's alignment improves or shifts over time.

Plagiarism Detection

Criterion integrates with MOSS (Measure Of Software Similarity, developed at Stanford) to detect code similarity across all student submissions for an assignment.

Running a Check

  1. Open the assignment detail page and navigate to the Plagiarism tab.
  2. Set a similarity threshold (default 70%). Pairs with a similarity percentage above this threshold are flagged in the results. Pairs below it are still shown but not highlighted.
  3. Click Run Plagiarism Check. Criterion collects all submissions and sends them to MOSS. Results typically appear within 2–5 minutes for a class of 50 students.

Reading the Results

Results appear as a ranked table of submission pairs:

  • Student A vs Student B — Names of the flagged pair.
  • Similarity %
    • 90%+ — High risk. Likely worth investigating.
    • 70–89% — Moderate. Review the MOSS report before drawing conclusions.
  • Matching lines — How many lines of code overlap.
  • MOSS report link — Opens the detailed MOSS side-by-side code comparison in a new tab. This shows exactly which sections of each submission match.
A high similarity score is evidence to investigate — not automatic proof of academic dishonesty. Common false positives include starter files, boilerplate code, and students who independently arrived at the same straightforward solution. Always review the MOSS report in full and consider context before taking action.

Re-Running Checks

You can run multiple checks on the same assignment, for example after regrading or after the submission deadline passes and all submissions are in. Each check creates a new report. Previous reports are preserved and accessible from the history list on the plagiarism tab.

Security & Privacy

This page explains how Criterion protects your data and what security measures are in place. If you are evaluating Criterion for institutional use, this section covers the questions your IT team is likely to ask.

Authentication & Sessions

  • Sessions use JWT tokens (HS256 algorithm, 32+ character secret).
  • Tokens are stored in HttpOnly session cookies — they are inaccessible to JavaScript, protecting against XSS token theft.
  • Sessions expire after 24 hours and must be re-authenticated.
  • Email verification is required before any account can log in.
  • Passwords are hashed with bcrypt — no plaintext passwords are ever stored.

Account Protection

  • Login lockout — After 5 consecutive failed login attempts, the account is locked for 30 minutes. This prevents brute-force attacks.
  • Rate limiting — Authentication endpoints are capped at 10 requests per minute per IP. General endpoints are capped at 10 requests per minute per user.
  • Password reset — Only via time-limited tokens sent to the registered email address. Tokens cannot be reused.

Data Protection

  • Canvas tokens encrypted at rest — When you connect your Canvas account, the API token is encrypted with Fernet symmetric encryption before being stored. The encryption key is separate from the database.
  • Soft deletes only — Business records (submissions, grades, comments, users) are never permanently deleted. The deleted_at field is set, but the row remains for audit purposes.
  • Immutable submission snapshots — A submitted workspace is frozen at submission time. The snapshot cannot be modified retroactively.
  • Audit logging — Every grading action (score change, comment, Canvas post, regrade, status update) is recorded with a timestamp, actor identity, and IP address.

Consent Management

  • All users must accept a data usage consent document before accessing the platform.
  • Consent documents are versioned. If the document is updated, users are required to re-accept on their next login.
  • Users who decline have their account deactivated. Consent acceptance is enforced by middleware on every protected route.

Network & Application Security

  • CORS — Cross-Origin Resource Sharing is restricted to the single approved frontend URL. API requests from other origins are rejected.
  • Content Security Policy (CSP) — Restrictive CSP headers are set on all API responses.
  • X-Frame-Options: DENY — Prevents clickjacking by disallowing Criterion pages to be embedded in iframes.
  • X-Content-Type-Options: nosniff — Prevents MIME-type sniffing.
  • X-XSS-Protection — Enabled.
  • HSTS — HTTP Strict Transport Security is enforced in production, requiring HTTPS for all connections.
  • Permissions-Policy — Camera, microphone, geolocation, and payment APIs are explicitly disabled.

Code Execution Sandbox

  • Student code runs in an isolated Judge0 sandbox, completely separated from the application server and database.
  • Sandboxes have no network access — student code cannot make outbound HTTP requests.
  • CPU time and memory are strictly limited per execution (configurable per assignment).
  • Execution results are fetched and stored by the backend — student code never touches the database directly.
