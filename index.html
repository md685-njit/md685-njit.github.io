Account Setup

Registration

Go to the Criterion login page and click Create an account. Fill in your full name, institutional email address, and a secure password (minimum 8 characters, must include a letter and a number). Select Instructor as your role.

  1. Enter your full name, email, and password.
  2. Select Instructor from the role dropdown.
  3. Submit the form. A verification email is sent to your address.
  4. Open the email and click the verification link.
  5. On first login you will be prompted to review and accept the data usage consent document. You must accept before accessing the platform.
Password reset is available at any time via the "Forgot password?" link on the login page. A reset link is sent to your registered email.

Connecting to Canvas

  1. From the instructor dashboard, click Connect Canvas Account.

  • Sync the full student roster with names and emails.

Healthy Stale Failed Syncing Paused Disconnected

Creating an Assignment

Click New Assignment from the course detail page. The assignment form is organized into five tabs. You can switch between tabs freely before saving.

Tab 1 — Basic Information

  • Title — The student-facing name (e.g., "Lab 3: Binary Search").
  • Description — Markdown-formatted assignment brief shown in the student workspace. Explain the problem, expected behavior, and any constraints.
  • Grading Instructions — Private notes visible only to instructors, TAs, and the AI grader. Use this to describe grading criteria, edge cases to watch for, or overall intent. Students never see this field.
  • Instruction File — Upload a PDF, Word, or text file (e.g., a detailed spec sheet). It appears as a pinned read-only link in the student's workspace sidebar.

Tab 2 — Starter Files

Provide skeleton code to pre-populate student workspaces. This helps students get oriented immediately without setting up boilerplate.

  • Upload one or more files using the file picker or drag-and-drop.
  • Create files manually by clicking New File and entering a name and content.
  • Preview and edit each file's content inline with syntax highlighting.
  • Remove or replace individual files as needed.
Starter files are copied into each student's workspace when they first open the assignment. Students can modify them freely.

Test Cases

Test cases define how student code is evaluated. Each test case has a weight that determines its share of the total score. The final score is calculated as:

score = (sum of weights of passed tests) / (sum of all weights) × points_possible

Criterion supports five test case types. Click each type below to see its fields.

stdin / stdout — Standard Input/Output Test

The most common test type. Criterion feeds a standard input to the student's program and compares the program's standard output to the expected output.

  • Name — A label shown in grading feedback (e.g., "Test 1: Empty list").
  • Stdin — Text input passed to the program via standard input. Upload a file for large inputs.
  • Expected stdout — The exact expected output. Upload a file or type it directly.
  • Float tolerance — Allowable numeric difference for floating-point outputs (e.g., 0.001).
  • Command-line arguments — Optional args passed when the program is invoked.
  • Weight — Relative importance (e.g., 1 for basic, 3 for edge cases).
  • Timeout — Per-test execution timeout, 1–60 seconds.
  • Visible to student — If checked, students see the test name, inputs, expected output, and their output. If unchecked, this test is hidden.
Trailing whitespace and newline differences are normalized before comparison, so minor formatting differences do not cause false failures.

Unit Test — Language-Specific Framework Tests

Write tests using the standard testing framework for each language. Criterion compiles and runs the test file alongside the student's code.

  • Test file — Upload or write the test code. Each language has its own format:
LanguageFrameworkRequirements
PythonpytestTest functions prefixed with test_
CCG_ASSERTCall cg_run_all_tests() in main()
C++CG_ASSERTCall cg_run_all_tests() in main()
JavaCgUnitTestMethods prefixed with test in a CgUnitTest class
JavaScriptJest-styledescribe/it/expect, import via require('./main.js')
Unit tests give students detailed pass/fail per assertion, helping them pinpoint exactly which function is broken.

Hidden Test — Secret Evaluation Cases

Hidden tests work identically to stdin/stdout tests but students never see the input, expected output, or actual output — only whether the test passed or failed.

  • Stdin — Input fed to the program (kept secret from students).
  • Expected stdout — Expected output (kept secret).
  • Float tolerance — Same as stdin/stdout.
  • Command-line arguments — Optional arguments (kept secret).
Use hidden tests to check edge cases, boundary conditions, and robustness without giving away test data. Students learn to write defensive code.

Script (Bash) — Custom Shell Judge

Write a bash script that receives the student's submission directory. The script performs any checks you define. An exit code of 0 means pass; any non-zero exit means fail.

# Example: verify output file exists and has content
if [ -s output.txt ]; then
  exit 0
else
  exit 1
fi

The script has full access to the student's files, can inspect file structure, run the program multiple times, or validate side effects.

Special Judge — Custom Python Validator

For problems with multiple valid answers (e.g., sorting algorithms with ties, approximate numeric outputs), write a Python 3 checker that awards a score fraction between 0.0 and 1.0.

import sys

expected_file = sys.argv[1]  # Path to expected output
actual_file   = sys.argv[2]  # Path to student's output
input_file    = sys.argv[3]  # Path to the test input

with open(actual_file) as f:
    actual = f.read().strip()

# Custom validation logic here
if is_valid(actual):
    print("1.0")   # Full credit
else:
    print("0.0")   # No credit

Print a value between 0.0 and 1.0 on the first line of stdout. This allows partial credit for partially-correct outputs.

Rubrics

Rubrics let you grade submissions on qualitative criteria — code quality, design, documentation — in addition to automated test results. The rubric editor is available from the assignment detail page after saving.

Creating a Rubric

  1. Open the assignment detail page and navigate to the Rubric section.
  2. Click Add Criterion to create a grading dimension (e.g., "Code Readability").
  3. Set the criterion's weight (relative importance among all criteria).
  4. Add 3–5 performance levels for each criterion:
    Each level has a label (e.g., "Excellent"), a score fraction (0.0–1.0), and a description of what earns that level.
  5. Repeat for each criterion. Weights are normalized automatically.
  6. Save the rubric and mark it as active to apply it to grading.

AI-Assisted Rubric Generation

Click Generate Rubric with AI and Criterion's AI will draft a complete rubric based on your assignment title, description, and grading instructions. Review and adjust the draft before saving.

How the Rubric Score Is Calculated

rubric_score = sum(criterion.weight × chosen_level.score_fraction for each criterion)
              / sum(criterion.weight for each criterion)

final_rubric_points = rubric_score × assignment.points_possible

The rubric score can be combined with or used independently from the automated test score depending on your grading workflow.

Assignment Settings

The Settings tab gives you fine-grained control over deadlines, scoring, late policies, Canvas linking, and the execution environment.

Dates & Points

  • Due Date — Timezone-aware deadline. Students see a countdown in their workspace.
  • Points Possible — Default 100. Determines the scale for test scores and rubric scores.
  • Max Attempts — Leave blank for unlimited. Set a number to cap how many times students can submit.

Submission Policy

  • Latest submission — Only the most recent submission counts for the grade.
  • Best effective score — The highest-scoring attempt (after late penalties) counts. Students can retry to improve.

Late Policy

PolicyBehavior
Hard CutoffNo submissions accepted after the deadline.
Grace PeriodAllow submissions for N minutes/hours after the deadline with no penalty. Configure the grace period length.

Canvas Linking

Search for the corresponding Canvas assignment by name and select it to link. Once linked, Criterion can post grades directly to that Canvas column. A "Canvas linked" badge confirms the connection.

TA Permissions

  • Enable Allow TAs to post grades to Canvas so TAs can confirm and post grades without instructor approval.
  • When disabled, TA reviews move to a "TA reviewed" state and require instructor confirmation before posting.

Runtime Environment

Override the default execution environment for specialized assignments:

  • Run timeout — Seconds allowed for the "Run" button in the student editor.
  • Test timeout — Seconds allowed per test case during grading.
  • Memory limit — Maximum memory (MB) per execution.
  • Compile command — Override the default compiler command with a custom bash script body.
  • Run command — Override how the compiled program is invoked.
  • Dependency manifest — Notes describing packages or libraries the grading sandbox should have available.
  • Hidden support files — Files injected into the sandbox that students cannot see (e.g., grading harness headers, fixture data files, helper libraries).

Canvas Deep Link

After saving an assignment, Criterion generates a unique deep link URL. When a student clicks this URL (typically embedded as a Canvas assignment link), they are automatically routed to the correct assignment workspace — no manual navigation required.

  1. Save the assignment. Criterion generates the deep link automatically.
  2. Copy the deep link URL from the assignment detail page.
  3. In Canvas, open the assignment and paste the URL into the submission type field (External Tool or Website URL).
  4. Students click the link in Canvas and are taken directly to their Criterion workspace.
If the student has never opened the assignment before, Criterion automatically creates a new workspace seeded with your starter files.

Running Grading Sessions

A grading session processes all student submissions for an assignment — compiling and executing each one against your test cases and generating AI feedback.

Starting a Batch Grading Session

  1. Open the assignment detail page.
  2. Click Start Grading Session.
  3. Criterion enqueues all pending submissions as background jobs.
  4. A real-time progress indicator shows completed students out of total (e.g., "14 / 47 completed").
  5. Individual results appear in the review queue as they complete.

What Happens During Grading

1

Snapshot Captured

The student's current workspace files are frozen into an immutable snapshot. This is the code that will be graded, regardless of any edits made after submission.

2

Code Execution

Each test case is compiled and executed in an isolated Judge0 sandbox. Results (pass/fail, stdout, stderr, time, memory) are recorded per test.

3

Score Calculated

The weighted sum of passing test cases is computed and converted to the assignment's point scale.

4

AI Feedback Generated

Claude AI produces 300–400 words of personalized, rubric-aligned feedback per student — covering what passed, what failed, and specific improvement suggestions.

5

Result Ready for Review

The result enters the review queue with status "pending review." You or a TA can now inspect it, adjust the score, add comments, and post to Canvas.

Regrading

If you update test cases or fix a grading issue, click Regrade All to re-execute every submission. You can also regrade an individual student's submission from the grading workspace.

Grading Workspace

The grading workspace is a three-panel interface where you review each student's submission, adjust scores, add comments, and post grades.

Left Panel — Review Queue

Students are organized into status buckets:

Ungraded Late Post Failed Needs Review Posted Graded

Filter by section or search by student name/email. Click a student to load their submission in the main panel. Use the arrow keys or previous/next buttons to move through the queue.

Main Panel — Submission Viewer

  • Code viewer — File tree on the left, syntax-highlighted code on the right. Browse all submitted files.
  • Test results — Per-test pass/fail icons with expandable detail rows showing stdout, expected output, and errors.
  • AI feedback — Generated student feedback displayed inline.
  • Rubric scoring — If an active rubric exists, select the appropriate performance level for each criterion. The rubric score updates in real time.

Right Panel — Grading Actions

  • Manual score override — Type a custom score to override the auto-graded result.
  • Feedback notes — Add text feedback shown to the student after posting.
  • Inline code comments — Click any line number in the code viewer to open a comment thread. Comments support replies and can be marked as resolved.
  • Flag for follow-up — Mark a submission as "needs review" with a reason note, visible only to graders.

Posting to Canvas

When you are satisfied with the review, click Post to Canvas. The grade is sent directly to the linked Canvas assignment column. The student sees their grade in Canvas immediately.

If the Canvas post fails (network error, token expiry), the status changes to "Post Failed" and you can retry from the queue without losing review data.

The Code Editor (Workspace)

The workspace is a browser-based IDE. No installation required — everything runs in your browser.

File Tree (Left Sidebar)

Submitting an Assignment

When you are ready, click Submit in the toolbar. A confirmation dialog shows how many attempts you have used and how many remain.

  1. Click Submit.
  2. Confirm in the dialog. Criterion captures a snapshot of all your current files.
  3. Your code is automatically graded in the background — this takes a few seconds to a minute depending on test complexity.
  4. The output panel updates in real time as each test case executes.
  5. When grading is complete, your score and test results appear.

Deadlines & Late Submissions

The header always shows the due date and time remaining. Depending on your instructor's late policy:

  • Hard cutoff — The Submit button is disabled after the deadline.
  • Grace period — You can submit for a short time after the deadline with no penalty.
  • Late penalty — You can submit anytime, but a percentage is deducted per day late. Your raw score and effective score (after penalty) are both shown.
Each submission uses one attempt. If your instructor set a max attempt limit, the attempt counter in the header shows how many you have left. Plan your submissions accordingly.

After your submission is graded, the bottom panel shows your results. What you see depends on the visibility setting your instructor chose:

AI Feedback

After grading, the AI feedback panel shows a 300–400 word analysis of your submission. It explains which tests passed, why some failed, and gives specific, actionable advice for improvement. This feedback is generated fresh for every submission.

Instructor Comments

If the submission policy is set to Best Score, your highest effective score across all attempts is the one that counts for your final grade.

Advanced Features

Tools for maintaining grading quality, academic integrity, and deeper pedagogical engagement.

Grader Calibration ▼

Calibration ensures that AI-generated grades and human graders are scoring consistently. It is especially valuable at the start of a course or after updating a rubric.

How Calibration Works

  1. Open the assignment detail page and click Calibration Workspace.
  2. Create a calibration session. Criterion automatically selects 5–10 diverse submissions as calibration samples.
  3. For each sample, review the student's code and the AI-generated grade, then enter your own instructor score.
  4. After scoring all samples, the system calculates the bias delta — the average difference between your scores and the AI's scores.
  5. Apply a correction factor to automatically adjust future AI grades, reducing systematic bias.

Calibration history is saved so you can compare correction factors across sessions and audit grading consistency over time.

Screenshot: Calibration workspace showing anchor scoring
Peer Review ▼

Peer review lets students evaluate each other's code using your rubric, providing additional feedback and encouraging critical thinking about code quality.

Configuring Peer Review

  • Reviews required — How many submissions each student must review (e.g., 2).
  • Allocation mode — Auto-random (shuffle across class), auto-section (within sections), or manual (instructor assigns pairs).
  • Anonymity mode
    • Blind — Reviewers and authors are both anonymous.
    • Reviewer hidden — Author sees code but not reviewer's name.
    • Named — Full identities visible.
  • Dates — Set when reviews open, the review deadline, and when reviews are released to reviewed students.
  • Visibility settings — Control whether reviewers see the code, test results, stdout, and AI feedback of the submission they are reviewing.

Student Reviewer Experience

Students access their review assignments from the My Reviews page. They see the assigned code (respecting visibility settings) and fill out a structured review form with rubric criterion scores, strengths, weaknesses, and suggestions.

Instructor Moderation

The peer review dashboard shows review completion counts, compliance metrics, and flagged reviews. Instructors can void problematic reviews or release all reviews to reviewees at once.

Plagiarism Detection ▼

Criterion integrates with MOSS (Measure Of Software Similarity, Stanford) to detect code similarity across all student submissions for an assignment.

Running a Check

  1. Open the assignment detail page and navigate to Plagiarism.
  2. Set a similarity threshold (default 70%). Pairs above this threshold are flagged.
  3. Click Run Plagiarism Check. Criterion collects all submissions and submits them to MOSS.
  4. Results appear once the check completes (typically a few minutes).

Reading the Results

Flagged pairs are shown in a table with:

  • Student 1 vs Student 2 — Names of the flagged pair.
  • Similarity % — Color-coded: 90%+ (high risk), 70–89% (moderate), below threshold (informational).
  • Matching lines — Number of similar lines detected.
  • MOSS report link — Opens the detailed MOSS side-by-side code comparison in a new tab.
A high similarity score is evidence to investigate, not automatic proof of academic dishonesty. Always review the MOSS report before taking action.
History & Audit Log ▼

The History browser gives instructors a complete, immutable record of all grading activity.

  • Session list — Every grading session ever run for your courses, with type (batch vs single), status, date/time, and student counts.
  • Session replay — Click any session to view the exact grades and test results as they were at the time of grading. Useful for grade disputes.
  • Grading lineage — If a student was re-graded multiple times, you can trace the full chain of results — who changed what, when, and from what value to what value.
  • Audit log entries — Every mutation (score change, comment add, Canvas post, regrade) is logged with a timestamp, actor identity, and IP address.

All audit data is retained permanently (soft-delete only — records are never hard-deleted).

Security & Privacy

Your data is protected

Criterion is built with institutional security requirements in mind. Here is exactly how your data is handled.

🔑 Authentication & Sessions

  • JWT tokens (HS256) with a 32+ character secret.
  • Sessions expire after 24 hours.
  • Tokens stored in HttpOnly cookies — inaccessible to JavaScript.
  • Email verification required before login.
  • Password hashed with bcrypt (no plaintext storage).

🔒 Account Protection

  • Account lockout after 5 failed login attempts (30-minute cooldown).
  • Rate limiting: 10 requests/minute per IP on auth endpoints.
  • Password reset via time-limited email tokens only.
  • Password requirements enforced: 8+ characters, letter + number.

💾 Data Protection

  • Canvas API tokens encrypted at rest using Fernet symmetric encryption.
  • All records use soft-delete — nothing is permanently destroyed.
  • Every grading mutation is logged in an immutable audit log.
  • Code submissions stored as immutable snapshots.

📝 Consent Management

  • All users must accept a data usage consent document before accessing the platform.
  • Consent documents are versioned — a new version requires re-acceptance.
  • Users who decline have their account deactivated.
  • Consent enforcement runs as middleware on every protected route.

🌐 Network Security

  • CORS restricted to the single approved frontend origin.
  • Content Security Policy (CSP) headers on all responses.
  • X-Frame-Options: DENY — prevents clickjacking.
  • HSTS enforced in production.
  • Permissions-Policy disables camera, microphone, geolocation, and payment APIs.

🥷 Code Execution Sandbox

  • Student code runs in an isolated Judge0 sandbox — completely separated from the application server.
  • Configurable CPU timeout and memory limits per test.
  • No network access from within the sandbox.
  • Execution results are fetched and stored; student code never touches the database directly.
